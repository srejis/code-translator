# Code Translator — 코드 단위 시각화 뷰어 구현 지시문

## 0. 작업 전제

대상 저장소는 `srejis/code-translator`이다.

현재 GitHub 원격 저장소에는 확인 가능한 파일과 브랜치가 없으므로, 실제 구현 전에 반드시 **현재 로컬 작업 폴더의 구조를 먼저 조사**한다. 이 지시문에 적힌 파일명이나 폴더명을 그대로 강제하지 말고, 이미 존재하는 프로젝트 구조·실행 방식·의존성 관리 방식에 맞춰 통합한다.

새로운 독립 프로젝트, 별도 ZIP 패키지, 중복 실행 진입점을 만들지 않는다. 기존 저장소 안에서 자연스럽게 동작하도록 구현한다.

---

## 1. 현재 상황

다음 명령으로 코드 분리 결과를 생성했다.

```bash
python segment_code.py D:\project\12_Gen_Code\codegraph-mvp --output backend_unit.json --batch-size 10
```

생성 결과:

```text
files=331
units=99673
parse_error_files=2
warnings=230
output=backend_unit.json
```

현재 문제는 `backend_unit.json`의 크기가 크고 구조가 복잡해 사람이 분리 결과를 검수하기 어렵다는 것이다.

---

## 2. 구현 목표

`backend_unit.json`을 IDE처럼 검수할 수 있는 로컬 시각화 프로그램을 구현한다.

핵심 화면은 다음 세 영역으로 구성한다.

1. 왼쪽: 파일 탐색기
2. 가운데: 원본 코드 뷰어
3. 오른쪽: 선택한 코드 단위의 상세 정보

가운데 코드 뷰어에서는 `segment_code.py`가 나눈 **각 코드 단위의 경계가 수평선과 거터 표시로 명확하게 보여야 한다.**

이 단계의 목적은 번역 기능 구현이 아니라 **LLM에 전달되기 전 코드 분리 결과를 사람이 검사하는 것**이다.

---

## 3. 가장 먼저 수행할 작업

코드를 수정하기 전에 다음을 확인한다.

1. 저장소 전체 폴더 구조
2. 현재 Python 패키지 구조
3. `segment_code.py`의 위치와 진입 방식
4. `requirements.txt`, `pyproject.toml`, `setup.cfg` 등 의존성 관리 파일
5. 기존 CLI 또는 `__main__.py` 존재 여부
6. `backend_unit.json`의 실제 최상위 구조와 필드명
7. 코드 단위 한 개의 실제 데이터 구조
8. 파일별 원본 코드가 JSON에 포함되는지, 외부 파일에서 읽어야 하는지
9. `warnings`, `parse_errors`, `batches`, `units`, `files` 등이 어떤 형태로 저장되는지

JSON 구조는 추정해서 구현하지 않는다. 실제 `backend_unit.json`의 앞부분과 대표 단위를 읽고 현재 스키마에 맞춰 구현한다.

기존 코드가 이미 뷰어·서버·CLI 구조를 일부 가지고 있다면 반드시 재사용한다.

## 4. 기술 조건

### 4.1 구현 언어

전체 구현은 Python을 기준으로 한다.

프런트엔드 화면에 필요한 HTML, CSS, JavaScript는 Python 서버가 제공하는 정적 자원으로 구성할 수 있다. 별도의 Node.js 프로젝트나 React 빌드 환경은 추가하지 않는다.

### 4.2 대용량 처리

`99,673`개 단위를 브라우저로 한 번에 전송하거나 DOM에 모두 렌더링하지 않는다.

다음 구조를 사용한다.

```text
backend_unit.json
→ 스트리밍 방식으로 읽기
→ SQLite 검색 인덱스 생성
→ 브라우저는 선택한 파일의 데이터만 요청
```

권장 사항:

- JSON 스트리밍에는 `ijson` 또는 현재 프로젝트에 이미 있는 동등한 도구를 사용한다.
- 첫 실행에서 SQLite 인덱스를 생성한다.
- 이후 실행에서는 원본 JSON의 수정 시간과 크기를 확인해 캐시를 재사용한다.
- JSON이 변경되었을 때만 인덱스를 다시 생성한다.
- 강제 재색인 옵션을 제공한다.
- SQLite 파일은 Git에 커밋하지 않도록 `.gitignore`에 포함한다.

전체 JSON을 `json.load()`로 메모리에 적재하는 방식은 금지한다.

### 4.3 로컬 전용

외부 서비스나 데이터베이스를 사용하지 않는다.

- 로컬 HTTP 서버
- 로컬 SQLite
- 로컬 파일 시스템

만 사용한다.

## 5. 데이터 인덱스 설계

실제 JSON 스키마를 확인한 뒤 다음 정보를 검색 가능하게 저장한다.

### 파일 정보

- 파일 경로
- 언어
- 코드 단위 개수
- 경고 개수
- 파싱 오류 여부
- 원본 코드 파일 존재 여부

### 코드 단위 정보

- `id`
- `parent_id`
- 파일 경로
- 시작 줄
- 종료 줄
- 노드 종류
- LLM 입력용 `code`
- 원본 범위 `raw_code`
- 경고
- 원본 순서
- 깊이 또는 부모 계층

실제 필드명이 다르면 현재 스키마에 맞춰 매핑 계층을 만든다. 뷰어 내부 데이터 모델과 JSON 원본 구조를 직접 결합하지 않는다.

## 6. 원본 코드 읽기 규칙

코드 뷰어에는 가능한 한 실제 원본 파일 전체를 표시한다.

우선순위:

1. 명령행으로 전달된 원본 저장소 경로에서 파일 읽기
2. JSON 메타데이터에 저장된 원본 루트 경로 사용
3. JSON에 파일 전체 내용이 있다면 해당 내용 사용
4. 위 방법이 모두 불가능하면 코드 단위의 `raw_code`를 이용한 제한 모드 표시

Windows 절대 경로와 상대 경로를 모두 처리해야 한다.

예상 실행 형태는 현재 프로젝트 CLI 구조에 맞추되, 최소한 다음 정보를 전달할 수 있어야 한다.

```text
입력 JSON 경로
원본 소스 저장소 경로
서버 포트
강제 재색인 여부
```

`backend_unit.json`이 `code-translator` 폴더에 있고 분석 대상이 다른 저장소인 상황을 지원해야 한다.

예:

```text
JSON: D:\project\14_code_translator\backend_unit.json
소스: D:\project\12_Gen_Code\codegraph-mvp
```

## 7. 화면 요구사항

## 7.1 전체 레이아웃

IDE와 유사한 3열 구조로 만든다.

```text
┌──────────────┬──────────────────────────────┬───────────────────┐
│ 파일 탐색기  │ 코드 뷰어                    │ 단위 상세 정보    │
│              │                              │                   │
│ 폴더/파일    │ 줄 번호 + 코드 + 경계선      │ ID / 범위 / 종류  │
│ 검색/필터    │                              │ code / raw_code   │
└──────────────┴──────────────────────────────┴───────────────────┘
```

각 열의 폭은 드래그로 조절할 수 있으면 좋지만, 필수 기능 완성 후 구현한다.

## 7.2 파일 탐색기

다음을 지원한다.

- 폴더 계층 펼치기·접기
- 파일 선택
- 경로 또는 파일명 검색
- 전체 파일 표시
- 경고가 있는 파일만 표시
- 파싱 오류가 있는 파일만 표시
- 언어별 필터
- 파일별 단위 수 표시
- 경고 및 파싱 오류 배지 표시

331개 파일을 처음부터 모두 펼치지 않는다.

## 7.3 코드 뷰어

다음을 표시한다.

- 줄 번호
- 원본 코드
- 코드 단위 시작 경계선
- 코드 단위 종료 경계선
- 거터의 단위 종류 또는 짧은 표시
- 선택된 단위 강조
- 부모·자식 단위 구분
- 경고 단위 표시

### 경계선 규칙

코드 단위가 시작되는 줄 위에 명확한 실선을 표시한다.

여러 줄 단위라면 종료 줄 아래에는 더 약한 선 또는 점선을 표시한다.

한 줄 단위라면 하나의 구획처럼 보이게 한다.

중첩 단위가 같은 줄에서 시작할 수 있으므로 단순히 선 하나만 그리지 말고, 다음 중 하나를 사용한다.

- 거터에 중첩 깊이 표시
- 같은 줄에서 시작하는 단위 목록 배지
- 클릭 시 해당 줄에서 시작하는 단위 선택 메뉴

단위의 경계가 겹칠 때 데이터를 삭제하거나 임의로 하나만 선택하지 않는다.

### 긴 파일 처리

선택된 파일의 코드가 길더라도 전체 줄을 한꺼번에 DOM에 만들지 않는다.

가상 스크롤을 사용하거나 현재 보이는 줄 주변만 렌더링한다.

스크롤 위치와 줄 번호가 안정적으로 유지되어야 한다.

## 7.4 단위 선택

다음 방법으로 단위를 선택할 수 있어야 한다.

- 코드 구획 클릭
- 거터 배지 클릭
- 단위 목록 클릭
- ID 검색

선택하면:

- 해당 코드 범위를 강조한다.
- 오른쪽 상세 패널을 갱신한다.
- 부모 단위와 자식 단위를 표시한다.
- 필요하면 해당 줄로 자동 스크롤한다.

## 7.5 오른쪽 상세 패널

선택한 단위의 다음 정보를 보여준다.

- 단위 ID
- 부모 ID
- 파일 경로
- 언어
- 노드 종류
- 시작 줄과 종료 줄
- LLM에 전달될 `code`
- 실제 원본 `raw_code`
- 경고 목록
- 부모 단위 링크
- 자식 단위 목록

`code`와 `raw_code`는 명확히 분리한다.

현재 단계에서는 번역 결과 편집기나 LLM 호출 버튼을 추가하지 않는다.

## 7.6 단위 목록 보기

파일별로 모든 단위를 순서대로 볼 수 있는 `Units` 보기 또는 탭을 제공한다.

각 행에는 다음을 표시한다.

- 순서
- 시작~종료 줄
- 노드 종류
- ID
- 부모 ID
- 코드 앞부분
- 경고 여부

목록도 가상 스크롤 또는 페이지 방식으로 처리한다.

## 8. 서버 API 요구사항

현재 프로젝트에 서버 구조가 없다면 최소한 다음 역할의 API를 만든다. 실제 URL과 모듈 배치는 기존 구조에 맞춘다.

### 메타데이터

- 전체 파일 수
- 전체 단위 수
- 경고 수
- 파싱 오류 파일 수
- 분석 대상 루트
- JSON 경로
- 인덱스 상태

### 파일 트리

- 폴더 및 파일 계층
- 파일별 단위 수
- 경고·오류 상태

### 파일 상세

- 파일 원본 코드
- 언어
- 해당 파일의 코드 단위 경계 정보

### 단위 목록

- 파일별 단위 목록
- 검색 및 페이지 처리

### 단위 상세

- 단위 데이터
- 부모 정보
- 자식 목록

API 응답은 필요한 데이터만 반환한다. 파일 하나를 열 때 전체 저장소 단위 데이터를 함께 반환하지 않는다.

## 10. 파싱 오류와 경고 처리

현재 파싱 오류 파일이 2개, 경고가 230개 존재한다.

이 정보가 뷰어에서 숨겨지면 안 된다.

- 파싱 오류 파일은 파일 트리에서 별도 배지로 표시한다.
- 오류 상세 내용을 확인할 수 있어야 한다.
- 경고 단위는 코드 거터와 상세 패널에 표시한다.
- 경고만 있는 파일을 필터링할 수 있어야 한다.
- 오류 때문에 전체 뷰어 실행이 중단되면 안 된다.

알 수 없는 JSON 항목이나 깨진 단위가 있으면 해당 항목을 건너뛰되 로그와 UI에 실패 이유를 남긴다.

## 11. 성능 기준

다음 조건을 목표로 한다.

- 첫 색인 이후 재실행 시 전체 JSON 재파싱 없이 시작
- 파일 트리 검색이 즉시 반응
- 파일 선택 시 해당 파일 데이터만 로드
- 10,000줄 이상 파일도 브라우저가 멈추지 않음
- 단위 99,673개를 브라우저 메모리에 한 번에 보관하지 않음
- 같은 파일을 다시 열 때 캐시 활용

속도를 위해 데이터 정확성을 희생하거나 단위 경계를 합치지 않는다.

## 12. 실행 방식

기존 CLI가 있으면 하위 명령으로 통합한다.

예시 개념:

```text
segment: 코드 단위 생성
view: 생성된 JSON 시각화
```

기존 CLI가 없다면 현재 저장소 구조에 가장 자연스러운 단일 진입점을 만든다.

사용자가 최소한 다음과 비슷한 방식으로 실행할 수 있어야 한다.

```bash
python <현재 프로젝트 진입점> view backend_unit.json --source-root D:\project\12_Gen_Code\codegraph-mvp
```

또는 현재 저장소의 실행 관례에 맞는 동등한 명령을 제공한다.

실행 후 기본 브라우저에서 로컬 주소를 자동으로 열되, 자동 실행에 실패하면 URL을 터미널에 출력한다.

서버는 `Ctrl+C`로 정상 종료되어야 한다.

## 13. 파일 배치 원칙

구현 전에 현재 폴더 구조를 확인하고 아래 원칙에 따라 배치한다.

- 기존 Python 패키지가 있으면 그 안에 뷰어 모듈 추가
- 기존 CLI 파일이 있으면 해당 CLI에 명령 추가
- 기존 웹 자원 폴더가 있으면 재사용
- 테스트 폴더가 있으면 그 안에 테스트 추가
- 문서는 기존 README에 추가하거나 프로젝트 문서 규칙에 맞는 위치에 작성
- 실행 결과 JSON, SQLite 캐시, 로그는 소스 코드 폴더와 분리

단순히 다음 파일들을 저장소 루트에 무조건 추가하는 방식은 피한다.

```text
code_unit_viewer.py
index.html
style.css
app.js
```

현재 구조상 루트 배치가 가장 적절하다고 확인된 경우에만 그렇게 한다.

## 14. 구현 범위에서 제외할 것

이번 작업에서는 다음을 구현하지 않는다.

- LLM 번역 호출
- 번역문 표시 또는 편집
- 사용자 계정
- 원격 서버 배포
- GitHub 저장소 직접 다운로드
- 코드 수정 기능
- IDE 수준의 자동 완성
- AST 재분석
- `segment_code.py` 분리 규칙 변경

이번 작업은 **현재 분리 결과를 정확하게 시각화하고 검수하는 기능**까지만 담당한다.

## 15. 테스트 요구사항

### 자동 테스트

최소한 다음 테스트를 추가한다.

1. 실제 스키마와 같은 작은 JSON의 색인 생성
2. 여러 파일과 중첩 `parent_id` 저장
3. Windows 경로 처리
4. JSON 변경 시 인덱스 재생성
5. 캐시 재사용
6. 파일별 단위 조회
7. 부모·자식 조회
8. 경고 및 파싱 오류 조회
9. 존재하지 않는 원본 파일 처리
10. 잘못된 JSON 항목이 있어도 전체 색인이 중단되지 않는지 확인

### 수동 검수

실제 `backend_unit.json`을 사용해 다음을 확인한다.

1. 331개 파일이 파일 트리에서 확인되는가
2. 총 단위 수가 원본 통계와 일치하는가
3. 파싱 오류 파일 2개가 표시되는가
4. 경고 230개가 확인 가능한가
5. 여러 줄 함수 입력값이 하나의 단위로 표시되는가
6. 함수 본문 내부 문장들이 각각 경계선으로 나뉘는가
7. 객체·배열 데이터의 분리 범위를 확인할 수 있는가
8. 콜백 부모와 내부 자식 단위가 연결되어 보이는가
9. 긴 TSX 또는 Python 파일에서 스크롤이 끊기지 않는가
10. 단위 클릭 시 오른쪽 상세 정보가 정확한가

## 16. 완료 조건

다음 조건을 모두 충족해야 완료로 본다.

- 로컬 저장소 구조를 먼저 조사하고 그 구조에 통합함
- `backend_unit.json` 전체를 색인할 수 있음
- 원본 코드 파일을 IDE 형태로 열 수 있음
- 파일별 코드 단위 경계가 선으로 명확히 표시됨
- 중첩 단위와 동일 시작 줄 단위를 잃지 않음
- 단위 상세 정보와 부모·자식 관계를 볼 수 있음
- 경고와 파싱 오류를 필터링할 수 있음
- 대용량 JSON과 긴 파일에서 브라우저가 멈추지 않음
- Windows 경로에서 실행됨
- 실행법과 캐시 동작이 문서화됨
- 기존 `segment_code.py` 동작을 깨뜨리지 않음
